## Видение проекта

Мы создаём **прототип нативной постквантовой защиты** для Solana на уровне ядра валидатора (Layer 0). Это не смарт-контракт и не программа — это модификация самого движка блокчейна, позволяющая ему нативно понимать и верифицировать подписи алгоритма **Falcon-512** (FN-DSA) наряду с классическим Ed25519.

Цель — не заменить Ed25519 полностью, а **добавить новый тип аккаунтов**: квантово-устойчивые "сейфы" для долгосрочного хранения крупных сумм.

---

## Почему это важно

### Квантовая угроза для Solana уникальна

В отличие от Bitcoin, где публичный ключ скрыт за хэшем (P2WPKH) до первой траты, **каждый адрес Solana — это сам публичный ключ Ed25519**. Это означает, что в Solana нет никакого "скрытого" слоя защиты. Как только появится квантовый компьютер, способный запускать алгоритм Шора, **все** существующие аккаунты Solana станут уязвимы — без исключений, без необходимости ждать чьей-то транзакции.

### Алгоритм Шора vs Алгоритм Гровера

| Алгоритм | Что атакует | Угроза для Solana |
|----------|-------------|-------------------|
| **Шора** | Эллиптические кривые (Ed25519, ECDSA, BLS) — позволяет вычислить приватный ключ из публичного | **Критическая** — ломает все подписи |
| **Гровера** | Хэш-функции (SHA-256) — квадратичное ускорение перебора | **Минимальная** — SHA-256 остаётся с 128-бит безопасностью |

Это означает, что PDA-аккаунты (основанные на SHA-256) остаются безопасными, а EOA-аккаунты (основанные на Ed25519) — нет.

### Временные рамки

Реальная квантовая угроза — это горизонт от 10 до 30+ лет (или никогда). Сегодняшние квантовые компьютеры оперируют сотнями "шумных" кубитов, тогда как для взлома Ed25519 нужны сотни тысяч — миллионы стабильных логических кубитов. Но **подготовка должна начинаться сейчас**, потому что миграция блокчейна на порядки сложнее миграции централизованных систем: каждый пользователь должен сам подписать транзакцию перевода на новый PQC-адрес, используя тот самый ключ, который становится уязвимым.

## Ключевая идея: Гибридная модель

Мы **не** пытаемся перевести всю Solana на PQC. Постквантовые подписи большие и медленные, что убивает главное преимущество Solana — пропускную способность. Вместо этого мы предлагаем **двухуровневую модель безопасности**:

```
┌─────────────────────────────────────────────────────┐
│                  PQC Vault (Falcon)                  │
│                                                     │
│  • Квантово-устойчивый "сейф"                       │
│  • Хранение крупных сумм (savings account)          │
│  • Редкие, простые транзакции (перевод на hot wallet)│
│  • Подпись: 666 байт, ключ: 897 байт                │
│                                                     │
│         ──── простой перевод SOL ────▶               │
│                                                     │
│                  Ed25519 Hot Wallet                   │
│                                                     │
│  • Стандартный быстрый кошелёк                      │
│  • Ежедневные операции: DeFi, NFT, стейкинг         │
│  • Сложные мульти-инструкционные транзакции          │
│  • Подпись: 64 байт, ключ: 32 байт                  │
└─────────────────────────────────────────────────────┘
```

### Пользовательский сценарий (User Flow)

1. Пользователь создаёт **PQC Vault** — генерирует Falcon-ключ, получает 32-байтный адрес (SHA-256 хэш от 897-байтного публичного ключа).
2. На PQC Vault хранится основной капитал — SOL, стейблкоины, ценные токены.
3. Когда нужно совершить торговлю на DEX или взаимодействие с DeFi:
   - Пользователь делает **одну простую PQC-транзакцию**: перевод нужной суммы с Vault → Hot Wallet.
   - С Hot Wallet совершает сколько угодно сложных транзакций (свопы, добавление ликвидности, минт NFT и т.д.).
4. Остаток возвращается на Vault обычным Ed25519-переводом.

### Почему это работает

- **PQC-транзакции тяжёлые** (~1564 байт только заголовок), но для простого `SystemProgram.transfer` этого достаточно.
- **Сложные DeFi-транзакции** (множество инструкций, CPI, несколько подписантов) остаются на Ed25519 и не теряют в производительности.
- Это **аналог cold storage + hot wallet** в традиционных финансах, но с криптографической квантовой защитой для "холодной" части.

## Почему Falcon (FN-DSA)

NIST стандартизировал (или находится в процессе) несколько PQC-схем подписей:

| Схема | Публичный ключ | Подпись | Статус NIST | Подходит для Solana? |
|-------|---------------|---------|-------------|---------------------|
| **Ed25519** (текущий) | 32 B | 64 B | — | Да (не PQC) |
| **ML-DSA** (Dilithium) | 1312 B | 2560 B | Стандарт | Нет — подпись 2560B не влезает в V1 транзакцию с полезной нагрузкой |
| **FN-DSA (Falcon-512)** | 897 B | 666 B | Черновик | **Да** — лучший баланс размер/безопасность |
| **SLH-DSA** (SPHINCS+) | 64 B | 7856 B | Стандарт | Нет — подпись 7856B превышает весь V1 лимит |
| **HAWK** | ~1024 B | ~555 B | Исследование | Потенциально, но не готов |

**Falcon-512 — единственный практичный выбор** для Solana прямо сейчас:
- Суммарный PQC-заголовок (magic + pubkey + sig) = **~1564 байт** — влезает в V1 транзакцию (4096 байт) с запасом для полезной нагрузки.
- ML-DSA потребовал бы ~3873 байт только на заголовок, оставляя ~223 байта — недостаточно даже для простого трансфера.
- 128-бит постквантовая безопасность (эквивалент сегодняшнего Ed25519).

> **Важно:** FN-DSA пока является черновиком NIST, а не утверждённым стандартом. Для production-решения это потребует дополнительной оценки. Для прототипа и демонстрации концепции — это оптимальный выбор.

## Карта квантовых угроз Solana

Не все аккаунты Solana одинаково уязвимы. Понимание этого — ключ к правильной расстановке приоритетов защиты.

### Что уязвимо (Ed25519)

| Тип аккаунта | Что хранит | Почему уязвим |
|-------------|-----------|---------------|
| **EOA (кошельки пользователей)** | SOL, токены | Адрес = публичный ключ Ed25519. Алгоритм Шора вычисляет приватный ключ → полный доступ к средствам |
| **Token Account authority** | Управление SPL-токенами (USDC, BONK и т.д.) | Authority — Ed25519 ключ владельца. Взлом ключа = кража всех токенов |
| **Upgrade Authority** | Право обновить код программы | Ed25519 ключ. Взлом → подмена кода программы → вывод всех средств из PDA |
| **Mint Authority** | Право выпускать новые токены | Ed25519 ключ. Взлом → бесконечная эмиссия токенов |
| **Freeze Authority** | Право заморозить токен-аккаунты | Ed25519 ключ. Взлом → блокировка средств пользователей |
| **Validator Identity / Vote Authority** | Управление валидатором, стейкинг | Ed25519 ключ. Взлом → манипуляция стейком и голосованием |
| **Stake / Withdraw Authority** | Управление стейкингом | Ed25519 ключ. Взлом → вывод застейканных SOL |

### Что уже безопасно (SHA-256 / нет приватного ключа)

| Тип аккаунта | Почему безопасен |
|-------------|-----------------|
| **PDA (Program Derived Addresses)** | Выводятся через SHA-256, **не имеют приватного ключа**. Подписать от имени PDA может только программа через `invoke_signed`. Алгоритм Шора бесполезен — нечего взламывать |
| **DeFi Vault'ы (пулы ликвидности, lending и т.д.)** | Это PDA. Основная масса TVL в DeFi (Raydium, Orca, Marginfi, Kamino) хранится в PDA-аккаунтах |
| **Immutable-программы** | Программы с отозванным upgrade authority не могут быть изменены никем, даже с квантовым компьютером |
| **Blockhash, Merkle Trees** | Основаны на SHA-256 — Гровер даёт лишь квадратичное ускорение, оставляя 128-бит безопасность |
| **Seed phrases** | Деривируют ключи через one-way KDF. Шор не может инвертировать хэш-функции. Квантовый атакующий может получить приватный ключ, но **не** seed phrase |

### Ключевой инсайт

Огромная часть TVL в DeFi-экосистеме Solana уже защищена от квантовых атак по дизайну — потому что средства лежат в PDA. Однако **административные ключи** протоколов (upgrade authority, mint authority) остаются уязвимы. Атакующий не сможет взломать PDA напрямую, но если он вычислит приватный ключ upgrade authority, он может **подменить код программы** на вредоносный и вывести средства через новую логику.

Это как непробиваемый сейф, ключ от которого висит на гвоздике.

**Наш PQC Vault решает проблему пользовательских кошельков.** Тот же подход применим и к authority-ключам протоколов: upgrade authority тоже может быть Falcon-адресом, защищая программу от квантовой подмены кода.

## Безопасность Hot Wallet: Exposure Window

Очевидный вопрос к гибридной модели: *"Деньги на Vault в безопасности, но в момент перевода на Hot Wallet они снова уязвимы."*

### Почему это не проблема

Квантовая атака на Ed25519 — это **не мгновенный взлом**. Даже самые оптимистичные оценки криптографов предполагают, что квантовому компьютеру потребуется **от нескольких часов до нескольких дней** на запуск алгоритма Шора против одного конкретного ключа. Это целенаправленная атака на один публичный ключ с огромными вычислительными затратами — не массовое сканирование.

```
Время жизни средств на Hot Wallet:  5–30 минут  (перевод → операции → возврат)
Время квантовой атаки на ключ:      часы — дни  (Шор против Ed25519)

                     ┌─────── Окно уязвимости ───────┐
─────────────────────┤ 5 мин                         ├──────────────────
                     └───────────────────────────────┘
─────────────────────┤ ~8 часов (оптимистичная оценка атаки) ├──────
```

Тратить часы квантовых вычислений (стоимостью в миллионы долларов) ради перехвата средств, которые исчезнут через 5 минут — **экономический абсурд**.

### Настоящая угроза — статичные адреса

Реальные цели квантового атакующего:
- Whale-кошелёк с 100 000 SOL, не двигавшийся 3 года
- Казначейство проекта с миллионами USDC на одном адресе
- Забытые кошельки ранних инвесторов

У этих целей есть два свойства: **большая сумма** и **бесконечное время на атаку**. Именно их и защищает PQC Vault.

### Harvest Now, Decrypt Later

Дополнительный вектор атаки: злоумышленник может **уже сейчас** записывать все транзакции блокчейна (он публичный). Через 20 лет, когда квантовый компьютер появится, он восстановит приватные ключи из публичных и украдёт средства с аккаунтов, которые к тому моменту всё ещё хранят активы.

PQC Vault защищает именно от этого сценария: даже если записать публичный Falcon-ключ сейчас, алгоритм Шора не способен вычислить из него приватный ключ ни сегодня, ни через 20 лет.

## Архитектура проекта

Проект состоит из двух независимых компонентов, взаимодействующих по сети:

```
┌──────────────────────────────┐         HTTP/UDP          ┌──────────────────────────────┐
│   Клиент (TypeScript/Node)   │ ◀──────────────────────▶  │   Валидатор (Rust / Agave)   │
│                              │     RPC :8899              │                              │
│  • Генерация Falcon-ключей   │     TPU :8000              │  • Приём V1 пакетов          │
│  • SHA-256 хэш → адрес      │                            │  • Парсинг magic byte        │
│  • Сборка V1 транзакции      │                            │  • Извлечение Falcon-ключа   │
│  • Подписание Falcon         │                            │  • SHA-256 → сверка с адресом│
│  • Отправка sendTransaction  │                            │  • Нативная Falcon-верифик.  │
│                              │                            │  • SVM исполнение транзакции  │
└──────────────────────────────┘                            └──────────────────────────────┘
       kit/ + demo-app/                                              agave/
```

### Компонент 1: Модифицированный валидатор Agave (Rust)

Модификации ядра Solana:

1. **Парсер транзакций** (`sdk/packet/`, `transaction-view/`) — распознаёт magic byte в заголовке подписи и читает 897B ключ + 666B подпись вместо стандартных 64B.
2. **Сетевой фильтр sigverify** (`perf/src/sigverify.rs`, `core/src/sigverify.rs`) — ядро проекта:
   - Извлекает Falcon-ключ из транзакции
   - Вычисляет SHA-256 хэш ключа
   - Сверяет хэш с адресом отправителя в `account_keys`
   - Вызывает нативную Falcon-верификацию
   - Устанавливает `is_signer = true` при успехе
3. **Зависимости** — крейт `pqcrypto-falcon` или `falcon-rust` в `Cargo.toml` модулей `perf` и `core`.

### Компонент 2: Модифицированный TypeScript SDK (kit)

Форк монорепозитория `anza-xyz/kit`:

1. **`@solana/keys`** — новая функция `generateFalconKeyPair()`, расширение `SignatureBytes` до 666B.
2. **`@solana/addresses`** — функция `getFalconAddressFromPublicKey()` (SHA-256 → 32B → Base58).
3. **Кодек транзакций** (`@solana/transactions`) — при обнаружении Falcon-подписи записывает в буфер: `[magic byte] + [897B pubkey] + [666B signature]`.

### Компонент 3: Демо-приложение

Скрипт `demo-app/demo.ts`:
1. Подключение к локальному валидатору
2. Генерация PQC Vault (Falcon) и Hot Wallet (Ed25519)
3. Airdrop SOL на PQC Vault
4. Перевод с PQC Vault → Hot Wallet (Falcon-подпись)
5. Перевод с Hot Wallet → другой адрес (Ed25519-подпись)

Это демонстрирует полный цикл гибридной модели.

## Как это работает

### Формат PQC-транзакции (V1, 4096 байт)

```
┌─────────────────────────────────────────────────────────────┐
│ Секция подписей                                             │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ [0xFF]  Magic Byte — "следующая подпись — Falcon"       │ │
│ │ [897B]  Публичный ключ Falcon-512                       │ │
│ │ [666B]  Подпись Falcon-512                              │ │
│ └─────────────────────────────────────────────────────────┘ │
│ Итого секция подписей: ~1564 байт                           │
├─────────────────────────────────────────────────────────────┤
│ Сообщение транзакции (~2500 байт доступно)                  │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ Header (num_signers, num_readonly, ...)                  │ │
│ │ Account Keys (32B × N)                                   │ │
│ │ Recent Blockhash (32B)                                   │ │
│ │ Instructions (program_id, accounts, data)                │ │
│ └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### Поток верификации на валидаторе

```
Транзакция V1 (4096B)
        │
        ▼
   TPU принимает пакет
        │
        ▼
   Парсер проверяет первый байт секции подписей
        │
        ├── 0xFF (magic byte) ──▶ PQC-путь:
        │                           1. Читает 897B → falcon_pubkey
        │                           2. Читает 666B → falcon_signature
        │                           3. SHA-256(falcon_pubkey) → 32B хэш
        │                           4. Сравнивает хэш с account_keys[0]
        │                           5. falcon_verify(message, signature, pubkey)
        │                           6. Успех → is_signer = true
        │
        └── Обычный байт ──▶ Ed25519-путь (без изменений)
        │
        ▼
   SVM исполняет транзакцию
```

### Деривация адреса

```
Falcon Public Key (897 байт)
        │
        ▼
   SHA-256 хэш
        │
        ▼
   32 байта (raw address)
        │
        ▼
   Base58 кодирование
        │
        ▼
   Solana-адрес PQC Vault (например: "7xKXtg2CW87...")
```

## Технические вызовы (решаемые)

### 1. Коллизия адресов с Ed25519 (Off-Curve Problem)

**Проблема**: SHA-256 хэш от Falcon-ключа может случайно оказаться валидной точкой на кривой Ed25519. Тогда существует приватный Ed25519-ключ для этого адреса, и злоумышленник теоретически мог бы подписать транзакцию от имени PQC-аккаунта обычной подписью.

**Решение**: Механизм, аналогичный PDA bump — итеративно добавлять salt/bump к хэшу и проверять, что результат **не лежит на кривой Ed25519**. Или зарезервировать фиксированный бит/байт в адресе, гарантирующий off-curve.

**Статус**: Требует реализации.

### 2. Бюджет байтов V1 транзакции

**Проблема**: PQC-заголовок занимает ~1564 из 4096 байт. Для одного подписанта остаётся ~2532B — достаточно для простых операций. Но два PQC-подписанта = ~3128B заголовка → менее 1000B на сообщение.

**Решение**: В рамках гибридной модели PQC-транзакции намеренно простые (перевод SOL). Сложные мульти-подписные операции остаются на Ed25519. Это не баг — это фича архитектуры.

### 3. Transaction ID

**Проблема**: Текущий txid в Solana — первая Ed25519-подпись (64B). С Falcon-подписью (666B) это нужно менять.

**Решение на уровне прототипа**: Валидатор генерирует txid внутренне. Для production потребуется переход на txid = hash(payload), что делает идентификатор signature-agnostic.

### 4. Производительность верификации Falcon

**Проблема**: Falcon-верификация медленнее Ed25519.

**Решение**: В гибридной модели PQC-транзакции редкие (единицы в день на пользователя), поэтому общий overhead на сеть минимален. Бенчмарки покажут конкретные цифры.

## Вызовы за пределами скоупа

Эти проблемы **не решаются** в нашем прототипе, но мы осознаём их существование и документируем для полноты картины.

### 1. Консенсус и голосование валидаторов (Votor)

**Суть**: В Alpenglow (новый консенсус Solana) валидаторы голосуют каждый слот (~400ms). Сейчас используется BLS-агрегация — 1000 подписей сжимаются в одну. **Постквантового аналога BLS-агрегации не существует.**

**Последствия**: Если заменить BLS на Falcon, каждый валидатор рассылает 666B подпись всем остальным каждые 400ms. При 1000+ валидаторах это создаёт лавину трафика.

**Исследования**: Lattice-based агрегация (Chipmunk), STARK-based aggregation — но ничего из этого пока не готово для production.

### 2. Распространение блоков (Rotor/Turbine)

**Суть**: Лидер разбивает блок на shreds (~1232B, размер MTU) и рассылает по дереву валидаторов. Сейчас каждый shred содержит Ed25519-подпись лидера (64B) для защиты от поддельных пакетов.

**Последствия**: Falcon-подпись (666B) занимает больше половины MTU, что катастрофически снижает полезную нагрузку.

**Возможные решения**:
- Одна подпись на FEC-set (набор shreds) с Merkle-доказательствами для каждого shred
- Одна подпись на весь блок с аутентифицированными каналами между валидаторами

### 3. Все Authority-ключи

**Суть**: В Solana все роли — это Ed25519 ключи: mint authority, freeze authority, upgrade authority, stake/withdraw authority, validator identity, vote authority. Полная PQC-миграция затрагивает каждую из этих ролей.

**Для нашего прототипа**: Мы фокусируемся только на EOA-аккаунтах (user wallets) и операции `SystemProgram.transfer`.

### 4. Массовая миграция пользователей

**Суть**: Для перевода средств на PQC-адрес пользователь должен подписать транзакцию **текущим** Ed25519-ключом — тем самым, который становится уязвимым. Это гонка со временем: если квантовый компьютер появится до завершения миграции, незащищённые аккаунты будут скомпрометированы.

**Для нашего прототипа**: Мы создаём новые PQC-аккаунты "с нуля", не решая проблему миграции существующих.

### 5. Стандартизация FN-DSA

**Суть**: Falcon (FN-DSA) пока существует как **черновик** NIST, а не утверждённый стандарт. В будущем могут появиться более эффективные схемы (HAWK и другие).

**Наша позиция**: Архитектура с magic byte позволяет добавлять новые алгоритмы без переделки ядра — достаточно назначить новый magic byte для нового алгоритма.

### 6. Кошельки и экосистема

**Суть**: Ни один существующий кошелёк (Phantom, Solflare, Backpack) не поддерживает Falcon-ключи. Для реального использования потребуется:
- Интеграция PQC в кошельки
- Стандарт wallet-adapter для PQC
- Обновление всех SDK (Rust, Python, Go)
- Обновление эксплореров для отображения PQC-транзакций

## От локального прототипа к реальной сети

Наш прототип работает на локальном `solana-test-validator` (один узел). Ниже — честный анализ того, какие подсистемы Agave затрагивает наша модификация, а какие потребуют доработки для развёртывания на реальной сети с сотнями валидаторов.

### Что мы модифицируем

| Подсистема | Файл | Статус |
|-----------|------|--------|
| **TPU Sigverify** | `perf/src/sigverify.rs` | Модифицируем — основная точка входа PQC-верификации |
| **Core Sigverify** | `core/src/sigverify.rs` | Модифицируем — обёртка над perf sigverify |

Это точка входа транзакций в валидатор: TPU принимает пакет → `sigverify` проверяет подпись → банковский этап исполняет транзакцию. На локальном валидаторе этого **достаточно** для полного цикла.

### Что сломается на реальной сети

#### 1. Replay / Entry Verification — повторная проверка подписей

```
entry/src/entry.rs → UnverifiedSignatures::verify()
                   → signature.verify(pubkey, serialized_message)
```

Когда валидатор **воспроизводит блоки других лидеров**, он проверяет подписи заново — не через `sigverify.rs`, а через отдельный путь в `entry`. На локальном валидаторе вы сами лидер всех блоков, поэтому replay не вызывается. На реальной сети каждый валидатор должен уметь проверить PQC-подписи в чужих блоках.

**Необходимая доработка**: Добавить Falcon-верификацию в `entry/src/entry.rs` → `UnverifiedSignatures::verify()`.

#### 2. PoH Chaining — хэширование подписей в Proof of History

```
entry/src/entry.rs → hash_signatures()
                   → next_hash_with_signatures()
```

Подписи транзакций **вплетаются в цепочку Proof of History**. Этот код предполагает 64-байтные подписи (`Signature` = `[u8; 64]`). С 666-байтной подписью Falcon хэш PoH будет другим, и остальные валидаторы **не смогут воспроизвести** цепочку — консенсус развалится.

**Необходимая доработка**: Определить, как PQC-подписи участвуют в PoH (хэшировать полные 666B? или только 64B хэш от подписи?).

#### 3. Transaction ID — фундаментальный идентификатор транзакции

В Solana `txid = signatures[0]`, и это **хардкодно 64 байта** типа `Signature`. Используется повсюду:

| Где используется | Файл |
|-----------------|------|
| SVM Transaction trait | `svm-transaction/src/svm_transaction.rs` → `fn signature() -> &Signature` |
| Resolved Transaction View | `transaction-view/src/resolved_transaction_view.rs` → `self.view.signatures()[0]` |
| Geyser Plugin API | `geyser-plugin-interface/src/geyser_plugin_interface.rs` → "first signature, used for identifying the transaction" |
| Blockstore (RocksDB) | `ledger/src/blockstore/column.rs` → ключи `[u8; SIGNATURE_BYTES + ...]` |
| Bigtable (архив) | `storage-bigtable/src/lib.rs` → `get_bincode_cell("tx", signature.to_string())` |
| RPC subscriptions | `rpc/src/rpc_subscriptions.rs` → подписка по signature |

**Необходимая доработка**: Переход на `txid = SHA-256(transaction_payload)` — signature-agnostic идентификатор. Затрагивает десятки файлов.

#### 4. Тип `Signature` — фундаментальное ограничение

Тип `Signature` определён во **внешнем крейте** `solana-signature` (не в Agave) как ровно 64 байта. Это используется:

```
transaction-view/src/signature_frame.rs:
  MAX_SIGNATURES_PER_PACKET = PACKET_DATA_SIZE / (size_of::<Signature>() + size_of::<Pubkey>())
  // = 1232 / (64 + 32) = 12

perf/src/sigverify.rs:
  const MESSAGE_OFFSET: usize = 1 + size_of::<Signature>();  // = 65
```

Falcon-подпись (666B) не помещается в тип `Signature`. Наш обходной путь с magic byte работает на прототипе, но для production потребуется модификация базового типа в SDK.

**Необходимая доработка**: Новый тип `PqcSignature` или расширение `Signature` до enum `Ed25519(64B) | Falcon(666B)` в крейте `solana-signature`.

### Что НЕ затрагивается нашей модификацией (и не нужно менять)

Эти подсистемы работают с **валидаторными** ключами, а не с ключами пользователей. Наш PQC Vault касается только пользовательских транзакций, поэтому эти системы продолжают работать на Ed25519 без изменений:

| Подсистема | Файлы | Почему не затрагивается |
|-----------|-------|------------------------|
| **Gossip Protocol** | `gossip/src/protocol.rs`, `crds_value.rs`, `ping_pong.rs` | Межвалидаторная коммуникация — валидаторы подписывают gossip-сообщения своими Ed25519 identity-ключами |
| **Shred/Turbine Verification** | `ledger/src/sigverify_shreds.rs`, `turbine/src/sigverify_shreds.rs` | Лидер подписывает shreds своим Ed25519-ключом. PQC-подписи пользователей уже внутри транзакций, а не на уровне shreds |
| **Vote Verification** | `core/src/cluster_info_vote_listener.rs` | Валидаторы голосуют своими Ed25519 vote-ключами — отдельная тема от PQC-кошельков |
| **Repair Protocol** | `core/src/repair/serve_repair.rs` | Межвалидаторный протокол восстановления данных |
| **Ed25519 Precompile** | `precompiles/src/ed25519.rs` | On-chain программа для верификации Ed25519 подписей внутри транзакций — не путать с sigverify |

### Итоговая карта

```
                        Локальный test-validator     Реальная сеть
                        ────────────────────────     ──────────────
TPU Sigverify           ✅ Модифицировано             ✅ Работает
Replay/Entry Verify     ⚪ Не вызывается (1 узел)     ❌ Требует доработки
PoH Chaining            ⚪ Не критично (1 узел)       ❌ Требует доработки
Transaction ID          ⚪ Можно игнорировать          ❌ Требует переработки
Тип Signature (SDK)     ⚠️ Обходим magic byte          ❌ Требует расширения
Gossip                  ⚪ Не используется             ✅ Не затрагивается
Shred/Turbine           ⚪ Не используется             ✅ Не затрагивается
Vote Verification       ⚪ Формальное                  ✅ Не затрагивается
Repair Protocol         ⚪ Не используется             ✅ Не затрагивается
Ed25519 Precompile      ⚪ Не используется             ✅ Не затрагивается
```

**Вывод**: Для production-деплоя на реальную сеть потребуется доработка **4 подсистем** (replay, PoH, txid, SDK-типы). Инфраструктура валидаторов (gossip, turbine, votes, repair) продолжает работать на Ed25519 и не требует изменений — PQC Vault затрагивает только слой пользовательских транзакций.

## Сравнение с альтернативами

### Winternitz Vault (уже существует на Solana)

[solana-winternitz-vault](https://github.com/deanmlittle/solana-winternitz-vault) — смарт-контракт, использующий хэш-based one-time signatures (OTS).

| Параметр | Наш подход (Falcon Layer 0) | Winternitz Vault |
|----------|----------------------------|------------------|
| Уровень | Ядро валидатора (нативный) | Смарт-контракт (программа) |
| Тип подписи | Многоразовая (Falcon) | **Одноразовая** (Winternitz OTS) |
| UX | Как обычный кошелёк | Нужен новый ключ после каждой транзакции |
| Модификация ядра | Да (форк Agave) | Нет |
| Готовность | Прототип | Работает на mainnet |

**Ключевое преимущество нашего подхода**: Falcon-ключи переиспользуемые. Пользователь один раз создаёт PQC Vault и пользуется им годами. Winternitz требует генерации нового ключа после каждой подписи, что делает UX значительно сложнее.

### ML-DSA (Dilithium)

Стандарт NIST, но подпись 2560B делает его непрактичным для Solana даже с V1 транзакциями (4096B). Не остаётся места для полезной нагрузки.

### SLH-DSA (SPHINCS+)

Подпись 7856B — превышает весь лимит V1 транзакции. Невозможно использовать.
